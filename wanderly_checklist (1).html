⚠️ Weaknesses & Limitations

Honest Assessment
↓
ML Backend Not Connected
The Flask/scikit-learn backend exists separately but is not wired into the frontend. App currently runs on rule-based JS scoring in production, not a trained ML model.
↓
No Accuracy Evaluation Yet (MAE/RMSE)
Rubric requires ML evaluation metrics. No MAE, RMSE, or feature importance analysis has been run yet. Critical gap for final submission.
↓
LLM Content Accuracy Unverified
Flight prices, hotel names, restaurant names, safety scores, and currency rates are GPT-4o-mini estimates — not live data. Hallucination risk. No spot-checking done post-surgery.
↓
±10pt Random Jitter in Rankings
Non-deterministic results — same preferences can return different destinations on refresh. Intentional for diversity but undermines reproducibility. Should be replaced by ML-learned diversity.
↓
Hand-Tuned Scoring Weights
Budget 35% / Season 20% / Match 25% / Feasibility 10% were manually set — not learned from data. This is exactly what the scikit-learn model should replace.
↓
Base44 Vendor Lock-In & Credit Limits
Auth, database, and AI calls all tightly coupled to Base44 SDK. Credits ran out during development, affecting testing. Migration would require rewriting all three layers.
↓
No Live Flight/Hotel Price APIs
Booking links use deep links to external sites — Wanderly doesn't fetch actual live prices. Users see AI-estimated costs, not real-time data. Needs disclosure.
↓
Unknown Destination Scores Default to 75
When BrowseAndBudget generates a trip for a destination not in destinations.js, all scores default to 75 — a flat neutral baseline rather than a calculated score. ML backend would fix this.
